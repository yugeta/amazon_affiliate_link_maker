Chrome Extension 仕様整理
===

最終更新: 2026-07-23

## 1. 目的

この拡張機能の目的は、次の1点に限定する。

- ユーザーが開いている Amazon 商品ページから、アフィリエイトリンク生成に必要な情報を取り出し、テンプレート表示とコピー可能なHTMLを自動生成する。

## 2. 想定ユーザーフロー

1. ユーザーが Amazon 商品ページを開く
2. 拡張機能アイコンをクリックする
3. 開いているタブのURLを自動取得する
4. URLを入力欄に自動反映する
5. 生成処理を自動実行する
6. プレビューと生成HTMLを表示する

このフローでは、URL取得前に商品データ取得処理が走らないことを前提とする。

## 3. 仕様境界

### 3-1. 拡張機能で行うこと

- アクティブタブURLの取得
- 生成UIの表示
- 取得済みデータのテンプレート適用
- 生成HTMLの出力
- ユーザー設定の保持（storage）

### 3-2. 拡張機能で行わないこと

- src 側の実装都合をそのまま持ち込むこと
- 未確定URLで取得処理を開始すること
- 拡張の起動条件を外部HTML読込完了など別要因に依存させること

## 4. 現状の矛盾点と整理

### 4-1. 矛盾の本質

- 本来仕様は「AmazonページのURL取得が起点」だが、実装差分と同期差分により、URL取得以前に依存モジュールの読み込みエラーが起きる状態が発生した。
- src のデータ取得系（外部エンドポイントやフォールバック）の判断が拡張にも混在し、拡張本来の単純フローが崩れた。

### 4-2. 仕様としての解決方針

- 起点を固定する: 拡張起動 -> URL取得 -> 生成実行
- 依存関係を単純化する: URL取得と表示ロジックを優先し、取得実装を必要最小限に閉じる
- src と extensions の責務を分離する: src の実験/検証ロジックは拡張に直接持ち込まない

## 5. HTML取得方針に関する整理

質問の論点である「ブラウザで表示中なら、表示中HTMLを使えばよいのではないか」は仕様として妥当。

ただし実装では次を明確にする必要がある。

- 表示中HTMLを読む方式:
	- メリット: サーバー側クロール不要、起点が明確
	- 注意: content script 経由のDOM取得設計が必要

- サーバー側取得方式:
	- メリット: 取得処理をサーバーに集約できる
	- 注意: ネットワーク障害やレスポンス差異、仕様ドリフトの影響を受ける

この拡張の目的に対しては、まず「URL取得起点の一貫性」を守ることを優先し、取得方式は別途選択可能な設計にする。

## 6. 最小機能要件

- 起動時にURLを自動継承できる
- 継承URLで自動実行できる
- 生成結果（プレビュー/HTML）が表示される
- 例外時に処理が停止せず、理由が表示される

## 7. 連携対象（あなたの定義に沿った最小連携）

機能連携は以下に限定する。

- 取得HTMLをデータ分解する処理
- 分解結果をテンプレートへ反映する処理

それ以外（運用同期、コピー手順、srcの実験ロジック）は仕様外として分離する。

## 8. 実装責務マップ（現時点）

- URL自動継承起点:
	- extensions/contents/js/main.js
	- extensions/contents/js/amazon.js

- 生成実行:
	- extensions/contents/js/event.js
	- extensions/contents/js/create_crawl.js

- 表示と出力:
	- extensions/contents/js/data.js
	- extensions/contents/js/preview.js
	- extensions/contents/js/code.js
	- extensions/contents/js/template.js

## 9. 非機能要件

- 拡張起動後の初回反応を体感1秒以内にする
- 依存モジュール欠落で全体停止しない
- 同期時に拡張専用ファイルが壊れない

## 10. 今後の判断基準

次のルールに沿って変更可否を判断する。

- URL取得起点を壊す変更は不可
- 拡張専用責務を src 側事情で上書きする変更は不可
- 取得方式を変える場合でも、表示/テンプレート処理の契約は維持する

## 11. 現在のFix宣言

2026-07-23 時点で、extensions は次の仕様で Fix（基準版）とする。

- 取得フローは「アクティブタブHTML取得 -> HTML2data(JS) -> 表示/出力」。
- PHP API は extensions の主経路では使用しない。
- `extensions/contents/js/create_crawl.js` を基準実装とする。

### 11-1. 変更凍結ルール

- `extensions` 配下は、障害修正または仕様変更合意がある場合のみ変更する。
- src 側都合の同期・コピーで extensions の実装を上書きしない。
- 同期対象を追加する場合は、事前に `docs/extension.md` へ理由と影響範囲を追記する。

### 11-2. 解除条件

Fix状態の解除は次のいずれかを満たす場合のみ行う。

- 仕様変更（取得方式・表示契約・データ構造）が合意されたとき。
- Chrome拡張のAPI仕様変更で互換対応が必要なとき。
- Amazonページ構造変更で抽出ロジック更新が必要なとき。

## 付録: 旧メモ

Chromeウェブストア公開時のチェック項目や運用メモは、仕様本体と分離して管理する。
